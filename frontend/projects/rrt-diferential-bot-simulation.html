DifferentialBot RRT Simulator

Year: 2026

Overview

A 2D autonomous navigation simulator for a differential-drive robot built in PyGame and Pymunk. The robot explores a hallway-style map, detects obstacles with a cone-shaped ultrasonic sensor, constructs a live occupancy map, and navigates using a path computed by the Rapidly-exploring Random Tree (RRT) algorithm. A Kalman Filter fuses GPS and IMU readings into a smooth state estimate, and a heading-based PID controller translates each waypoint into precise left/right wheel speed commands. When the robot senses a close obstacle it backs up and triggers an automatic path replan — closing the full sense–plan–act loop. The code is structured for hardware portability, with commented-out MicroPython stubs (Raspberry Pi Pico + L298N) ready for real deployment.

Data flow — each simulation frame

1. Sense — UARTReceiver pulls fresh GPS coordinates, IMU angle & ultrasonic distance  ·  2. Estimate — KalmanFilter predicts & updates the state  ·  3. React — Controller checks proximity → backward motion + replan if needed  ·  4. Plan — RRT_Search generates a collision-free path from current position to goal  ·  5. Control — PIDController computes left/right speeds toward next waypoint  ·  6. Act — WheelMotorDrivers applies commands to the Pymunk body  ·  7. Map — Mapping records new detections into the occupancy grid

Algorithm Deep Dive

RRT — Rapidly-exploring Random Tree

The planner incrementally builds a tree rooted at the robot's current grid position. At each iteration it samples a random point in map space, finds the nearest existing node, and steers toward it by distance_at_node grid units. A new node is added only if the segment is collision-free (not in Map.obstacles_coordinates). Once a node lands within 2 grid units of the goal, the path is extracted by traversing parent pointers back to the root — then smoothed before being handed to the controller.

  • Max iterations — 3 000 (configurable in RRT_Search.py)
  • Step size — 2 grid units ≈ 20 px (set in Control.py)
  • Replanning — triggered automatically when the ultrasonic sensor detects an obstacle closer than the threshold
  • Collision check — queries the live occupancy map so the planner respects obstacles discovered at runtime

Kalman Filter — Sensor Fusion

A 4-state Kalman Filter (x, y, vx, vy) runs every frame using a constant-velocity motion model. The predict step propagates the state forward; the update step fuses GPS (position) and IMU (velocity/angle) measurements with separate, tunable noise covariances. The filtered estimate is used by the controller and rendered as a second trail on screen for visual comparison against raw GPS.

  • Process variance — 1e-4 (model confidence)
  • GPS measurement variance — 1e-2
  • IMU measurement variance — 1e-2

PID Heading Controller

The controller computes the atan2 angle to the next RRT waypoint and compares it with the robot's current heading. The error drives a bang-bang + full-speed strategy: if error > π/15 rad the robot spins left; if error < −π/15 rad it spins right; otherwise it drives straight at full speed. This keeps the robot tightly aligned to each waypoint without oscillation in the physics engine.

Hardware Deployment — Raspberry Pi Pico

The codebase is structured for real-hardware portability. Both WheelMotorDrivers.py and UARTReceiver.py contain commented-out MicroPython blocks for Raspberry Pi Pico GPIO control via an L298N motor driver (PWM pins). To deploy: (1) uncomment the from machine import Pin block in WheelMotorDrivers.py, (2) implement real UART reads in UARTReceiver.py's refresh_data() method, and (3) replace sensor simulation calls with actual hardware reads in SensorModules.py.
